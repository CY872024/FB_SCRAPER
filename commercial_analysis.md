# FB Scraper SaaS 商業價值與營運分析報告

本報告針對「自動化 Facebook 貼文監控與推播系統」進行商業化潛力評估，分析其目標客群、獲利模式、營運成本及潛在風險。

---

## 1. 核心商業價值與痛點解決 (Value Proposition)

目前 Facebook 的演算法經常「隱藏」或「延遲」粉絲專頁的發文，導致使用者錯過重要資訊。本系統解決了以下痛點：
* **去演算法化**：保證 100% 捕獲目標粉專的新貼文，不受 FB 觸及率下降影響。
* **搶佔時間差**：(每 15 分鐘監控) 對於限量、限時的活動報名，提供毫秒必爭的優勢。
* **無程式碼 (No-code) 門檻**：將複雜的 Python 爬蟲包裝成極簡的手機 Web App，降低一般使用者的技術門檻。

## 2. 目標客群分析 (Target Audience)

這套系統的潛在付費用戶非常明確，可分為三大類：
1. **行銷公關與社群小編 (B2B)**：
   * **需求**：監控競品粉專的最新活動、公關危機處理、社群風向監測。
2. **活動愛好者 / 搶票族 (B2C)**：
   * **需求**：博物館導覽報名、演唱會讓票社團、限量商品發售通知。
3. **業務開發與代購業者 (B2B/B2C)**：
   * **需求**：監控特定社團內的「求購文」、房仲監控地區社團的「求租文」以第一時間私訊客戶。

## 3. 建議的付費訂閱機制 (Pricing Strategy)

SaaS 模式最適合採用「分級訂閱制 (Freemium)」，以功能和頻率來區分付費階級：

| 方案 | 月費定價 (預估) | 排程頻率限制 | 任務數量上限 | 監控粉專上限 | 適合對象 |
| :--- | :--- | :--- | :--- | :--- | :--- |
| **免費版 (Free)** | $0 / 月 | 僅限「每天定時」1 次 | 1 個任務 | 3 個粉專 | 體驗用戶、一般民眾 |
| **專業版 (Pro)** | $9.99 / 月 | 開放「每 15 分鐘」即時監控 | 5 個任務 | 20 個粉專 | 搶票族、重度活動愛好者 |
| **企業版 (Biz)** | $29.99 / 月 | 開放「每 15 分鐘」即時監控 | 無限制 | 無限制 (提供多組收件人) | 行銷公司、房仲、業務團隊 |

> [!TIP]
> **獲利試算**：若能累積 100 位 Pro 用戶與 20 位 Biz 用戶，每月被動收入可達約 $1,600 美金 (約 5 萬台幣)。

## 4. 營運成本分析 (Cost Analysis)

目前的架構是「極致的零成本 (Zero-cost)」，但若走向商業化，成本結構將發生改變：

### 目前階段 (0 到 100 名會員)
* **主機 (GitHub Actions)**：$0 (Public Repo 無限免費，但若濫用可能違反 GitHub 條款)。
* **資料庫 (Firebase Standard)**：$0 (每月 5 萬次讀取完全夠用)。
* **發信 (Gmail SMTP)**：$0 (每天上限 500 封信)。

### 商業化擴張階段 (100 名以上付費會員)
一旦開始收費，必須將基礎設施移出免費層，以確保系統穩定性與 SLA (服務級別協議)：
* **運算主機 (VPS 或雲端函數)**：不能再依賴 GitHub Actions，需租用 DigitalOcean VPS 或 AWS EC2 (每月約 $20~$50 美金)。
* **專業發信服務 (SendGrid / Mailgun)**：避免 Gmail 被封鎖，需購買專門發信服務 (每月約 $15 美金，可發數萬封)。
* **防封鎖代理伺服器 (Residential Proxies)**：**(最大隱藏成本)** 爬蟲流量變大後，會被 FB 封鎖 IP，需購買代理 IP 服務 (每月約 $50~$100 美金)。

## 5. 潛在風險與營運挑戰 (Risks & Mitigations)

> [!WARNING]
> **1. Facebook 阻擋爬蟲 (Anti-Scraping)**
> * **風險**：FB 經常更改網頁結構 (DOM)，或對高頻率造訪的 IP 彈出驗證碼 (CAPTCHA)。
> * **對策**：需持續維護 Playwright 的抓取邏輯 (目前程式碼已採用智慧模擬下拉與動態等待)。未來需導入 Proxy 輪替機制。

> [!CAUTION]
> **2. GitHub 服務條款 (TOS) 風險**
> * **風險**：GitHub Actions 的條款明文規定，禁止使用免費的 CI/CD 資源進行「與軟體開發無關的商業爬蟲或加密貨幣挖礦」。如果流量過大，您的 GitHub 帳號可能會被停權。
> * **對策**：在測試與小規模自用階段無妨。一旦開始向使用者**收費**，請務必將 Python 程式碼搬移到自己租用的 VPS 主機 (例如 Linode 或 GCP) 上設定 Cronjob 執行。

> [!NOTE]
> **3. 隱私與法規 (GDPR/PDPA)**
> * **風險**：收集了用戶的 Email 與 Facebook 監控名單。
> * **對策**：在網頁加上隱私權條款，聲明系統不會儲存任何使用者的 FB 密碼 (系統是無登入爬取公開貼文)，確保免責。

## 💡 總結建議

這套系統具備**極高的商業潛力**，因為它解決了極具痛點的「資訊落差」問題。

**下一步行動建議：**
1. **MVP 試營運**：維持目前的 0 成本架構，先私下邀請 5~10 位有需求的朋友或同業免費試用，收集反饋。
2. **驗證剛需**：觀察他們最常設定的「關鍵字」與「粉專」是什麼類型？（這決定了您未來的行銷方向）。
3. **主機轉移**：當決定開始收費的那一天，立刻花 5~10 美金租用一台低階 VPS 伺服器，將 Python 爬蟲移植過去，正式建立自己的護城河。
